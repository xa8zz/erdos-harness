# Add round doc

## When to use

User pastes researcher/verifier output and wants it saved as a properly-formatted round doc. Also use when the user says "add this to the repo" or similar after any research result lands.

## Purpose

Combine two steps into one: (a) byte-faithful extraction from the session transcript, (b) front-matter composition based on in-session context. Result: a round doc that's immediately indexable by the compile script.

## Instructions

1. Pick a **unique marker phrase** visible in the user's paste (5–10 words, present nowhere else in the session).
2. Decide on the round doc's **front-matter fields** from conversation context:
   - `id` — next available `R<NN>-<slug>`. Use the round number from context; slug is a short kebab-case description.
   - `type` — `research` | `verification` | `refutation` | `synthesis` | `diagnostic`.
   - `date` — today in YYYY-MM-DD.
   - `intent` — 1–2 sentences stating what the round aimed to answer.
   - `predecessors` — list of prior round ids this builds on.
   - `action.kind` — `standalone` by default; `refutes` / `supersedes` / `extends` / `confirms` when the response overturns or builds on a specific prior round (then set `action.target`).
   - `claim` — the specific mathematical statement the round asserts.
   - `failure_mechanism` — **required for `type: refutation`**. Specific arithmetic/combinatorial feature that killed the target claim, with constants.
   - Optional: `implications`, `verifiers_at_time`, `confidence_at_time`, `strategy_dependence`.
   - `prompt` — optional repo-relative path to the prompt doc that produced this round. Skip if unknown or not applicable; include it whenever the prompt exists as a file.
3. **Do not** include a `status` field — status is derived by `compile_rounds.py` from the action graph.
4. Compose the YAML front-matter and pipe it into `scripts/add_round_doc.py` via stdin:

```bash
./scripts/add_round_doc.py \
  --marker "<exact phrase from paste>" \
  --target erdos-872/researcher-<NN>-<type>-<slug>.md <<'YAML'
id: R<NN>-<slug>
type: refutation
date: 2026-MM-DD
intent: >
  <1-2 sentences>
predecessors: [R<NN-prev>-<slug>]
action:
  kind: refutes
  target: R<NN-target>-<slug>
claim: >
  <specific mathematical statement>
failure_mechanism: >
  <specific arithmetic/combinatorial feature, with constants>
confidence_at_time: high
YAML
```

5. If the paste has multiple labeled blocks (#1, #2, #3), run the script once per block using `--split-on '^#\d+\s*$'` and `--split-index 0/1/2/...` with different `--target` paths and front-matter for each.
6. After writing, run `./scripts/compile_rounds.py --root <problem-dir>/ --out <problem-dir>/state_compiled.md` to regenerate the state view.

## Output

A new round doc at `--target`. No need to edit `current_state.md` by hand — `compile_rounds.py` regenerates the state view from all round docs.

## Gotchas

- **Never include a `status` field** in front-matter. Status is always derived. Writing one would mutate state at write-time, which is the exact thing this system prevents.
- **Front-matter is immutable.** If a later round overturns this one, add a new round with `action.kind=refutes` pointing at this id — do not edit this file.
- **Marker must be unique in the session.** If it matches an earlier entry, you'll extract the wrong content. Use a 5–10 word phrase that's distinctive.
- **Branched sessions create new jsonl files.** The script already handles this (sorts by mtime, searches all), but if extraction returns nothing, check that the paste is actually in the freshest session.
- **For `type: refutation`, `failure_mechanism` is required.** It's the pattern-across-failures signal. Include specific primes, shield sets, lemmas, constants — not "X didn't work."
- **No session jargon in `intent` or `claim`**: no "Round N", "prior round", "our program", "Pro", "Codex". Write for a reader who has no repo context. This is the same discipline as researcher prompts — the round doc itself should read as self-contained mathematical content.

## Example (refutation round)

```bash
./scripts/add_round_doc.py \
  --marker "Prolonger plays an even element of U and preempts separators" \
  --target erdos-872/researcher-56-pro-tau-sf-refuted.md <<'YAML'
id: R56-tau-sf-refuted
type: refutation
date: 2026-04-19
intent: >
  Determine whether tau_SF (separator-first Shortener) can achieve O(r_1(n))
  as proposed in R52.
predecessors: [R52-separator-first-hypothesis, R54-sigma-star-audit, R55-hidden-packet-collapse]
action:
  kind: refutes
  target: R52-separator-first-hypothesis
claim: >
  tau_SF fails as a full-game Shortener strategy. Prolonger forces
  L >= cn = omega(r_1(n)) via upper-half dyadic shielding.
failure_mechanism: >
  Prolonger plays an even element of U = (n/2, n], then preempts each
  separator s_Q with 2^a * s_Q in U before Shortener can play it. Total
  cost: n^(1/8+o(1)) separator moves + O(n/log n) prime-block moves =
  o(r_1). After this phase, >= cn composites in U are untouched but
  tau_SF has no legal move.
implications:
  - Any separator-first Shortener needs a composite fallback rule to reach r_1.
  - Narrows remaining attack space: pure-separator strategies exhausted.
verifiers_at_time:
  - [claude-R56-audit, agrees]
  - [gemini-R56-audit, agrees]
confidence_at_time: high
strategy_dependence: tau-SF-specific
YAML
```
